{% comment %} Publish / Re-publish confirmation dialog — CSP-clean, no inline JS. Publishing pushes OSV/CSAF (and the CVE record) to the public repo, so the operator must confirm the *identity* of what they're publishing by entering the advisory's ECL-… ID before the action is allowed. To keep it from being cumbersome, the ID is shown as a click-to-copy chip (data-copy, handled by static/advisoryhub-clipboard.js) right above the field — copy, then paste. The submit button is gated client-side: it ships `disabled` and is enabled only once the typed/pasted value matches `data-confirm-token` (the `data-confirm-submit` gate in static/advisoryhub-forms.js). The value is re-checked server-side in publication.views.publish — the gate is convenience, not the security boundary. Modelled on the hand-written forget-user dialog (admin_console/user_detail.html) and the self-contained branch of _confirm_action.html (which can't carry input fields). Params: dialog_id required — unique per page; also seeds the heading id. action required — POST target (the publish URL). advisory_id required — the ECL-… ID to copy and confirm against. trigger_label required — text on the opening button. confirm_label required — text on the confirming submit button. title required — dialog heading. body required — one-line explanation of the consequence. trigger_class optional — semantic variant for the trigger button. confirm_class optional — semantic variant for the confirm button. {% endcomment %}
