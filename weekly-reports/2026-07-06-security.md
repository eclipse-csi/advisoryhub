## Headline

**The application's security posture is strong and the fix this week is a single insecure default, not a
vulnerability in the code.** Authorization is server-side and consistent (no IDOR found — every
`get_object_or_404` is followed by a `can_*` check); `owner` is structurally underivable
(`access/models.py` omits it from `Permission.choices`, rejected at API/service/view boundaries —
INV-AUTH-3); notifications re-authorize every recipient at send time via `perms.can_view`
(`notifications/recipients.py:41-54`, INV-AUTH-1); OIDC groups are DB-mirrored on every login and
never trusted from the client (INV-OIDC-1/2); step-up proves `auth_time` from the ID token rather than a
session flag (`accounts/step_up.py:122-150`); Celery is JSON-only (no pickle); the GHSA webhook verifies
HMAC-SHA256 with `hmac.compare_digest` *before* parsing; comments are nh3-sanitized
(`comments/services.py`); the publication git layer blocks path traversal (`is_relative_to` +
`core.symlinks=false`, INV-PUB-8) and never shells out with `shell=True`; and secrets are funnelled
through `redact_secrets` on every persisted error (INV-SECRET-1..3). CSP is nonce-strict `strict-dynamic`
and **enforced** by default; cookies carry `__Host-` prefixes and `Secure`/`HSTS` in prod.

**The outlier is an insecure default.** `config/settings/base.py:18` defaults `DJANGO_SECRET_KEY` to the
literal `"insecure-change-me"`, `base.py:211` assigns it, and `config/settings/prod.py` had **no guard**
rejecting it. A production deploy that forgets to set the variable boots with a *publicly known signing
key* — forgeable session cookies, CSRF tokens, and signed password/login tokens = auth bypass. Django's
`security.W009` only *warns*, only under `check --deploy`, and CI runs that advisory-only (`|| true`)
under **test** settings (`.github/workflows/ci.yml:101`) — so nothing fails the actual boot. The Helm
chart requires the Secret; a bare-metal / docker-compose-prod deploy does not. This week's fix makes
`prod.py` **fail closed** at import time.

**Continuity — this closes an already-identified backlog item.** The same-day dependencies review flagged
this as **dependencies-F6** ("consider prod.py hard-fail on the sentinel default (spec-affecting)") and
deferred it pending maintainer sign-off on the startup/deploy-behavior change. The approved plan for this
review *is* that sign-off, so it is implemented here from the security lens. Its sibling
**dependencies-F4** (`CACHE_URL` unset → per-process rate limits + fragmented ALTCHA replay protection) is
the natural next companion and is carried forward below (SEC-2), not re-fixed this week.

## Fixed this week

- **SEC-1 — Reject the development-default `DJANGO_SECRET_KEY` at production startup.**
  Branch `fix/prod-secret-key-guard` — **local only, not pushed / no PR, per maintainer instruction.**
  - `config/settings/_guards.py` (new): dependency-light module (imports only
    `django.core.exceptions`) holding the sentinel constant `INSECURE_SECRET_KEY_DEFAULT` and
    `require_production_secret_key(secret_key)`, which raises `ImproperlyConfigured` when the key is
    empty or equals the dev default.
  - `config/settings/base.py`: the env() default now references `INSECURE_SECRET_KEY_DEFAULT` (line 18)
    instead of a duplicated literal — single source of truth, so the sentinel and the guard can never
    drift apart (a drift would silently make the guard fail *open*).
  - `config/settings/prod.py`: calls `require_production_secret_key(SECRET_KEY)` immediately after
    `DEBUG = False`, before anything else in the module. Runs at settings-import time, i.e. at gunicorn
    boot — the reliable "fail closed" point (a Django system check would only fire under `manage.py
    check`, which a forgetful bare-metal deploy also skips).
  - `common/test_prod_settings.py` (new, 3 tests): exercises the pure guard directly (no DB, no module
    reload) — raises for `""` and the sentinel, passes for a real 50-char key.
  - `.env.example` (line 16) and `docs/specification/architecture.md` §7.3 env-var inventory updated in
    the same commit (spec-lockstep rule); no new `INV-*` ID minted — this strengthens existing
    secret-hygiene intent rather than changing behavior the spec defines.
  - **Why it matters:** the well-known key is in the public repo; anyone can forge
    `django.core.signing` values (sessions, CSRF, password-reset/login tokens). This is the classic
    "hard-coded/default secret in production" weakness (CWE-798/CWE-1188). The guard converts a silent
    misconfiguration into a loud, immediate boot failure.
  - **Risk/rollback:** the *only* behavior change is that a prod process which relied on the insecure
    default now refuses to start — the intended, correct outcome, and a config fix (set the env var),
    not a code fix. Zero blast radius for dev/test/CI: they run under `config.settings.test`, and the
    guard lives only in `prod.py`. Rollback = revert the commit; no migrations, no data, no API change.

weighted* form as I=3 (Helm masks it); this review scores the *security impact* (I=5) because the whole
point of the fix is to make the misconfiguration impossible. Both agree it is worth fixing; the rubric
score (25) puts it first under strict prioritization, and the effort proved to be S (done this week).

No P0 found: no live code path leaks data, mis-authorizes, or corrupts state. SEC-1 is a latent
deploy-time footgun, now closed.

## Recommended order of work

1. **SEC-2** (S/M) — the direct sibling of SEC-1. Add a prod-only guard/system check that requires a real
   `CACHE_URL` (or explicitly opts into LocMem) so rate limits + ALTCHA replay protection aren't silently
   per-worker outside Helm. Wants maintainer sign-off (spec change) exactly as SEC-1 did — the
   `require_production_secret_key` pattern in `config/settings/_guards.py` is the template to extend.
2. **SEC-5** (S) — add the missing `review_status ∈ ReviewStatus.choices` check; mechanical, mirrors the
   `state` block directly above. Bundle into any `api/` change.
3. **SEC-3** (M) — install a `logging.Filter` that runs `redact_secrets` over records on the app/worker
   loggers, closing the raw-`log.exception` path centrally rather than per call site.
4. **SEC-4** (M) — a scheme+host allowlist (or reuse of `common/net.py`) for the three outbound targets,
   as defense-in-depth even though they're config-sourced today.
5. **SEC-7**, then **SEC-6** opportunistically.

## Not fixing yet — and why

- **SEC-2 (`CACHE_URL`):** deliberately *not* bundled with SEC-1. Both are "fail loudly outside Helm"
  hardening, but combining two startup-behavior guards in one commit violates the focused-change
  constraint and muddies the spec-lockstep diff. It is the clean next-week item and reuses SEC-1's guard
  module.
- **SEC-3 (log redaction filter):** the safe fix is a central logging filter, not sprinkling
  `redact_secrets` at each `log.exception`; that's a cross-cutting change deserving its own commit + tests.
- **SEC-4 (SSRF allowlist):** the URLs are settings-sourced, so exploitability requires a deployment
  compromise that already implies bigger problems. Real but low; not worth rushing an allowlist design.
- **SEC-8 / SEC-10:** policy/ops posture, not code defects — flagged for operator awareness, not patched.
